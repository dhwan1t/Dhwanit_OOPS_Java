Prompt 0-A · Scaffold the project
I am building a Course Management System in Java for a university lab viva. My existing code has these classes (package: Uni.OOPS_Lab.com.course): - model.java: static inner class Course (courseId, courseName, maxSeats, enrolledStudents with getters/setters + display()) and inner class Student (studentId, studentName) - exception.java: CourseFullException and CourseNotFoundException - service.java: CourseService with addCourse(), enrollStudent() writing to file.txt, and viewCourses() reading from file.txt - main.java: CLI demo I need to extend this into a full system with JDBC (SQLite), Servlet, JSP, Swing, and an Applet — all kept as SIMPLE as possible for a viva. Architecture: UI (Swing/JSP) → Servlet → Service Layer → JDBC → SQLite Do NOT change model.java or exception.java. Start by creating the full project folder structure and a build plan listing exactly which files to create.

JDBC — Database Layer
JDBC
▾
Prompt 1-A · DBConnection utility
Create a DBConnection.java file in the package Uni.OOPS_Lab.com.course. Requirements: - Use SQLite via the sqlite-jdbc driver (Class.forName("org.sqlite.JDBC")) - Database file: "courses.db" in the project root - Provide a static getConnection() method returning java.sql.Connection - Keep it minimal — just connection management, no extra logic Also write the SQL CREATE TABLE statement for a Course table: CREATE TABLE IF NOT EXISTS Course ( id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT NOT NULL, maxSeats INTEGER, enrolled INTEGER DEFAULT 0 ); Show where and how this table creation SQL gets called (hint: in DBConnection or a DBInit class).
Prompt 1-B · Update service.java for JDBC
Update service.java (CourseService class) to replace the file-based storage with JDBC/SQLite. Keep the same public method signatures so nothing else breaks: - addCourse(model.Course c) — INSERT into Course table - viewCourses() — SELECT all from Course table, print to console - getAllCourses() — SELECT all, return List<model.Course> (new method, needed by Servlet/Swing) Use DBConnection.getConnection() for all DB access. Use try-with-resources for Connection, PreparedStatement, ResultSet. Map DB columns → model.Course fields: id→courseId, name→courseName, maxSeats→maxSeats, enrolled→enrolledStudents. Keep exception handling simple — just print stack trace on SQLException. Do NOT change the class/method names. Do NOT use any ORM or external library beyond sqlite-jdbc.
Viva tip: Be ready to explain PreparedStatement vs Statement — say "PreparedStatement prevents SQL injection and is pre-compiled".

Servlet — Web Controller
Servlet
▾
Prompt 2-A · CourseServlet.java
Create CourseServlet.java in package Uni.OOPS_Lab.com.course. This single servlet handles both adding and viewing courses: doPost (add course): - Read parameters: courseName, maxSeats from request - Create a model.Course object and call CourseService.addCourse() - Redirect to GET (PRG pattern): response.sendRedirect("CourseServlet") doGet (view courses): - Call CourseService.getAllCourses() to get List<model.Course> - Set it as a request attribute: request.setAttribute("courses", courseList) - Forward to view.jsp: request.getRequestDispatcher("view.jsp").forward(...) Servlet mapping URL: /CourseServlet Use @WebServlet("/CourseServlet") annotation (no need for web.xml mapping). Import: javax.servlet.*, javax.servlet.http.*, java.util.List Keep the servlet under 50 lines. No business logic in the servlet — delegate everything to CourseService.
Viva tip: Explain the POST→Redirect→GET pattern: "It prevents duplicate form submissions on browser refresh."
Prompt 2-B · web.xml
Create WEB-INF/web.xml for the Course Management web app. Include: - welcome-file: index.jsp - servlet declaration and mapping for CourseServlet (as a fallback in case @WebServlet is not picked up) - Standard servlet 4.0 XML header Keep it minimal — just the essentials needed to deploy on Apache Tomcat 9+.

JSP — Web UI Pages
JSP
▾
Prompt 3-A · index.jsp (Add Course form)
Create index.jsp — a simple HTML form to add a course. Requirements: - Form action="CourseServlet" method="POST" - Two input fields: courseName (text) and maxSeats (number) - A submit button labeled "Add Course" - A link/button to navigate to view.jsp or CourseServlet GET to see all courses - Basic inline CSS — clean, readable, no external CSS framework needed - No scriptlets (<% %>) in this file — it's just a static HTML form wrapped in JSP Keep it under 40 lines total. Plain and viva-friendly.
Prompt 3-B · view.jsp (Display courses)
Create view.jsp — displays all courses from the database. Requirements: - Use JSTL core tag: <%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %> - The servlet puts the course list as request attribute "courses" (List<model.Course>) - Use <c:forEach items="${courses}" var="course"> to loop and show each course in a table - Table columns: Course ID, Course Name, Max Seats, Enrolled Students - Show a "No courses found" message if list is empty using <c:if test="${empty courses}"> - A "Back / Add Course" link pointing to index.jsp - Basic inline CSS table styling only Use JSTL — no Java scriptlets. Keep it clean and simple.
Viva tip: If asked why JSTL: "JSTL separates Java logic from HTML presentation, following MVC — the View should not contain business logic."

Swing — Desktop GUI
Swing
▾
Prompt 4-A · CourseSwing.java
Create CourseSwing.java in package Uni.OOPS_Lab.com.course — a simple Swing desktop GUI. Layout: - JFrame titled "Course Management System" - Top panel: JTextField for course name, JTextField for max seats, JButton "Add Course" - Center: JTable displaying all courses (columns: ID, Name, Max Seats, Enrolled) - Bottom: JButton "Refresh / View Courses" Behavior: - On "Add Course" click: read fields, create model.Course, call CourseService.addCourse(), then refresh table - On "Refresh" click: call CourseService.getAllCourses(), update DefaultTableModel - Use SwingUtilities.invokeLater to launch on EDT CourseService should be instantiated once as a field. The DB + JDBC is already handled by CourseService — Swing just calls it. Keep it simple: no MVC within Swing, just one class, one JFrame, straightforward ActionListeners. Under 100 lines.
Viva tip: "Swing is a GUI widget toolkit for Java. JFrame is the window, JPanel organizes components, and ActionListener handles button events. I used SwingUtilities.invokeLater to keep GUI updates on the Event Dispatch Thread."
